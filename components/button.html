          Button

          Buttons are used to trigger user actions within the interface.

          Overview

          Buttons represent explicit actions a user can take. They are interactive, intentional and result in an immediate outcome, such as submitting a form, saving changes or progressing a task.

          Buttons should be used sparingly and deliberately. Overuse reduces their effectiveness and makes it harder for users to identify primary actions.

          When to use

          Use buttons when the user is expected to perform an action.

          • Submitting or confirming an action
          • Saving or applying changes
          • Progressing through a workflow
          • Triggering an interface update or system behaviour

          When not to use

          Do not use buttons for navigation or passive interaction.

          • Navigating between pages or views (use links)
          • Inline text actions within body content
          • Decorative emphasis
          • Actions that do not result in a clear outcome

          Structure

          Buttons are built using semantic button elements.

          A button consists of:

          • A clickable button element
          • A text label describing the action
          • Optional supporting elements such as icons

          Buttons must always include visible text unless explicitly designed as icon-only buttons with an accessible label.

          States and behaviour

          Buttons support a predictable set of states.

          • Default: The button is available for interaction
          • Hover: Visual feedback on pointer interaction
          • Focus: Clear indication of keyboard focus
          • Active: Feedback during activation
          • Disabled: The action is unavailable

          State changes should reinforce affordance and availability without altering the meaning of the action.

          Variants

          Buttons support a limited set of variants to express hierarchy, emphasis and context.
          Variants affect appearance and emphasis, not behaviour.

          Emphasis variants

          Emphasis variants communicate the relative importance of an action.

          Primary button

          Used for the main or most important action on a page or within a section. 

          <button class="btn btn--primary" type="button">
  Primary action
</button>

          Most screens should contain only one primary action.

          Secondary button

          Used for supporting actions that are less prominent than the primary action. 

          <button class="btn btn--secondary" type="button">
  Secondary action
</button>

          Secondary buttons should not visually compete with the primary action.

          Ghost (low-emphasis) button

          Used for tertiary or low-priority actions where minimal visual emphasis is required. 

          <button class="btn btn--ghost" type="button">
  Tertiary action
</button>

          Ghost buttons must remain clearly identifiable as interactive.

          Size variants

          Size variants support different content densities and interaction contexts.
          Size affects padding and text size only, not importance.

          Large

          Used where actions require greater emphasis or touch comfort. 

          <button class="btn btn--primary btn--lg" type="button">
  Large button
</button>

          Medium (default)

          Used in most interface contexts.

          <button class="btn btn--primary" type="button">
  Medium button
</button>

          Small

          Used in dense interfaces or constrained spaces.

          <button class="btn btn--primary btn--sm" type="button">
  Small button
</button>

          Small buttons should not be used for primary page-level actions.

          Icon buttons

          Icon buttons are used where space is limited or when the action is universally recognisable.

          Icon with visible label

          <button class="btn btn--secondary btn--icon" type="button">
  <i class="fa-solid fa-download" aria-hidden="true"></i>
  <span>Download</span>
</button>

          Icons should support the label, not replace it.

          Icon-only button (hidden label)

          Icon-only buttons must provide an accessible name.

          <button
  class="btn btn--ghost btn--icon-only"
  type="button"
  aria-label="Close"
>
  <i class="fa-solid fa-xmark" aria-hidden="true"></i>
</button>

          Icon-only buttons should be used sparingly and only where the meaning is clear without text.

          Disabled state

          Disabled buttons indicate unavailable actions and must not be interactive. 

          <button class="btn btn--primary" type="button" disabled>
  Action unavailable
</button>

          Disabled buttons must not receive focus and should clearly appear inactive.

          Accessibility

          Buttons must be operable using both pointer and keyboard input.

          • Buttons must be focusable and operable via keyboard
          • Focus indicators must be clearly visible
          • Disabled buttons must not receive focus
          • Labels must clearly describe the action

          Buttons must not rely on colour alone to communicate importance or state.

          Content guidelines

          Button labels should be concise and action-oriented.

          • Use clear verbs that describe the outcome
          • Avoid vague labels such as “OK” or “Click here”
          • Keep labels short and scannable

          Button text should describe what will happen when the button is activated.

          Summary

          Buttons provide clear, predictable triggers for user actions.
          They should be used intentionally to support interaction and decision-making.

          Buttons clearly indicate available actions, express hierarchy through consistent variants, and provide predictable interaction feedback.
